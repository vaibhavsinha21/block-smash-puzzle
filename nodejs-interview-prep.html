Core Node.js Concepts

Must Know
1 What is Node.js and how does it work?
Must Know
▼

Node.js is an open-source, cross-platform JavaScript runtime built on Chrome's V8 engine. It uses an event-driven, non-blocking I/O model that makes it lightweight and efficient for building scalable network applications.

How it works:

  • V8 Engine — Compiles JavaScript into machine code (written in C++)
  • Event Loop — Single-threaded loop that handles async operations
  • libuv — C library providing thread pool for heavy I/O (file system, DNS, crypto)
  • Non-blocking I/O — Operations are delegated and handled via callbacks/promises when complete
💡 Interview Tip
Mention that Node.js is NOT a framework or language — it's a runtime environment.
2 Why is Node.js single-threaded? How does it handle concurrency?
Must Know
▼

Node.js uses a single main thread for JavaScript execution to keep things lightweight and avoid the overhead of thread context switching. However it achieves concurrency through:

  • Event Loop — Processes callbacks and events asynchronously
  • libuv Thread Pool — Handles heavy I/O in background (default 4 threads, configurable via UV_THREADPOOL_SIZE)
  • Worker Threads — For CPU-intensive tasks within the same process
  • Cluster Module — Spawns multiple processes across CPU cores
JSconst crypto = require("crypto");
const start = Date.now();

function logHashTime() {
  crypto.pbkdf2("a", "b", 100000, 512, "sha512", () => {
    console.log("Hash:", Date.now() - start, "ms");
  });
}

// All 4 run concurrently via the thread pool!
logHashTime(); // Hash: ~1200ms
logHashTime(); // Hash: ~1210ms
logHashTime(); // Hash: ~1205ms
logHashTime(); // Hash: ~1215ms
3 What is libuv? What role does it play?
Intermediate
▼

libuv is a multi-platform C library that provides:

  • Event Loop implementation
  • Thread Pool (4 threads by default) for file system, DNS, crypto
  • Async I/O — networking (TCP/UDP/DNS), file operations
  • Signal handling, child processes, timers

It's the bridge between Node.js JavaScript code and the operating system.

7 Explain the Event Loop phases in detail
Must Know Advanced
▼
┌───────────────────────────┐ ┌─>│ TIMERS │ ← setTimeout, setInterval │ └───────────┬───────────────┘ │ ┌───────────┴───────────────┐ │ │ PENDING CALLBACKS │ ← I/O callbacks deferred │ └───────────┬───────────────┘ │ ┌───────────┴───────────────┐ │ │ IDLE / PREPARE │ ← internal use │ └───────────┬───────────────┘ │ ┌───────────┴───────────────┐ │ │ POLL │ ← retrieve new I/O events │ └───────────┬───────────────┘ │ ┌───────────┴───────────────┐ │ │ CHECK │ ← setImmediate() │ └───────────┬───────────────┘ │ ┌───────────┴───────────────┐ │ │ CLOSE CALLBACKS │ ← socket.on('close') │ └───────────┬───────────────┘ │ │ └──────────────┘ ⚡ Microtask Queues (run BETWEEN each phase): 1. process.nextTick() queue ← highest priority 2. Promise .then() queue

Key Rules:

  • Microtasks (nextTick + Promises) run between every phase
  • process.nextTick() has higher priority than Promises
  • setImmediate() runs in the check phase
  • Timers have a minimum delay (even setTimeout(fn, 0) is ~1ms)

9 What causes Event Loop blocking? How do you detect it?
Advanced Amex
▼

Common causes:

  • Large JSON JSON.parse() / JSON.stringify() on big payloads
  • Heavy regex (ReDoS — Regular Expression Denial of Service)
  • Synchronous file operations (fs.readFileSync)
  • CPU-intensive loops (sorting, crypto without async)
  • Tight polling / infinite loops

Detection:

  • Monitor event loop lag using perf_hooks module
  • Use clinic.js for flamegraphs and profiling
  • APM tools (Datadog, New Relic) track event loop delay
  • --prof flag for V8 CPU profiling
JS// Measure event loop delay
const { monitorEventLoopDelay } = require('node:perf_hooks');
const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();

setInterval(() => {
  console.log(`Event Loop Delay — p50: ${h.percentile(50)}ms, p99: ${h.percentile(99)}ms`);
}, 5000);

  try {

    const orders = await getOrders(user.id);

    return details;
  } catch (err) {
    logger.error('Failed to fetch order details', err);
    throw err; // re-throw for caller
  }
}

13 What is an EventEmitter? How does it work?

EventEmitter is a core class implementing the Observer Pattern. Objects emit named events that cause registered listeners to be called.

JSconst EventEmitter = require('node:events');

class PaymentProcessor extends EventEmitter {
  processPayment(amount) {
    // ... process payment logic
    this.emit('payment:success', { amount, id: 'txn_123' });
    // or: this.emit('payment:failed', { error: 'Insufficient funds' });
  }
}

const processor = new PaymentProcessor();

processor.on('payment:success', (data) => {
  console.log(`Payment of $${data.amount} successful`);

});

processor.on('payment:failed', (data) => {

processor.processPayment(100);
⚠️ Memory Leak Warning

14 What is middleware in Express? How does next() work?
Must Know
▼

Middleware is a function that has access to (req, res, next). It can:

  • Execute any code
  • Modify request/response objects
  • End the request-response cycle
  • Call next() to pass control to the next middleware
JS// Custom middleware example
const requestLogger = (req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    logger.info(`${req.method} ${req.originalUrl} ${res.statusCode} ${duration}ms`);
  });
  next(); // MUST call next() or send a response!
};

// Error-handling middleware (4 params — MUST have all 4!)
const errorHandler = (err, req, res, next) => {
  logger.error(err.stack);
  res.status(err.statusCode || 500).json({
    status: 'error',
    message: err.isOperational ? err.message : 'Internal Server Error'
  });
};

// Order matters!
app.use(requestLogger);       // 1st: logging
app.use(express.json());      // 2nd: body parsing
app.use('/api', authMiddleware); // 3rd: auth (specific routes)
app.use('/api', router);       // 4th: routes
app.use(errorHandler);         // LAST: error handler

17 "Cannot set headers after they are sent" — What causes this?
Advanced
▼

This error occurs when you try to send more than one response for a single request.

Common causes:

  • Calling res.send() / res.json() and then continuing execution without return
  • Calling next() after already sending a response
  • Multiple async branches resolving and sending responses
JS// ❌ BUG — Missing return!
app.get('/user', (req, res) => {
  if (!req.user) {
    res.status(401).json({ error: 'Unauthorized' });
    // Execution continues! 💥
  }
  res.json(req.user); // Second response — CRASH!
});

// ✅ FIX — Always return after sending a response
app.get('/user', (req, res) => {
  if (!req.user) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  res.json(req.user);
});

19 How do you handle JWT authentication securely?
Must Know Amex
▼
  • Sign with RS256 (asymmetric) for microservices, HS256 for monolith
  • Short access token (15 min) + refresh token (7 days)
  • Store in HttpOnly, Secure, SameSite=Strict cookies (NOT localStorage!)
  • Implement token revocation via Redis blacklist
  • Always validate iss, aud, exp claims
JSconst jwt = require('jsonwebtoken');

// Generate tokens
function generateTokens(user) {
  const accessToken = jwt.sign(
    { id: user.id, role: user.role },
    process.env.JWT_SECRET,
    { expiresIn: '15m', issuer: 'amex-api' }
  );
  const refreshToken = jwt.sign(
    { id: user.id },
    process.env.REFRESH_SECRET,
    { expiresIn: '7d' }
  );
  return { accessToken, refreshToken };
}

// Auth middleware
function authenticate(req, res, next) {
  const token = req.cookies.accessToken; // HttpOnly cookie
  if (!token) return res.status(401).json({ error: 'No token' });

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch (err) {
    return res.status(403).json({ error: 'Invalid token' });
  }
}

▼

23 How do you diagnose memory leaks in production?
Advanced Amex
▼

Step-by-step approach:

  1. Monitor process.memoryUsage() over time — if heapUsed keeps growing, there's a leak
  2. Take heap snapshots at intervals using v8.writeHeapSnapshot()
  3. Compare snapshots in Chrome DevTools to find growing retained objects
  4. Common culprits:
    • Unbounded caches (Maps/arrays that grow forever)
    • Unremoved event listeners
    • Closures holding onto large objects
    • Global variables accumulating data
    • Unreleased DB connection pool handles
  5. Tools: clinic.js, 0x (flamegraphs), Chrome DevTools (remote debug), APM tools
JS// Quick memory monitoring endpoint
app.get('/health/memory', (req, res) => {
  const mem = process.memoryUsage();
  res.json({
    rss: `${Math.round(mem.rss / 1024 / 1024)}MB`,
    heapUsed: `${Math.round(mem.heapUsed / 1024 / 1024)}MB`,
    heapTotal: `${Math.round(mem.heapTotal / 1024 / 1024)}MB`,
    external: `${Math.round(mem.external / 1024 / 1024)}MB`
  });
});
24 How do you implement graceful shutdown?
Must Know Amex
▼ 
JSconst server = app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});

function gracefulShutdown(signal) {
  console.log(`${signal} received. Starting graceful shutdown...`);

  // 1. Stop accepting new connections
  server.close(() => {
    console.log('HTTP server closed');

    // 2. Close DB connections
    mongoose.connection.close(false, () => {
      console.log('MongoDB connection closed');

      // 3. Close Redis
      redisClient.quit(() => {
        console.log('Redis connection closed');
        process.exit(0);
      });
    });
  });

  // 4. Force close after timeout
  setTimeout(() => {
    console.error('Forced shutdown — timeout exceeded');
    process.exit(1);
  }, 30000);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT', () => gracefulShutdown('SIGINT'));
🏦 Why Amex cares
In financial services, dropping requests = dropped transactions = lost money. Graceful shutdown ensures zero-downtime deployments.

27 What is backpressure in streams? What happens if you ignore it?
Advanced
▼

Backpressure occurs when a writable stream can't consume data as fast as the readable stream produces it.

If ignored: Data buffers in memory → memory bloat → OOM crash

Solution: .pipe() handles this automatically. When writing manually:

JSconst readable = fs.createReadStream('large.csv');
const writable = fs.createWriteStream('output.csv');

readable.on('data', (chunk) => {
  const canContinue = writable.write(chunk);
  if (!canContinue) {
    readable.pause(); // Stop reading until drain
  }
});

writable.on('drain', () => {
  readable.resume(); // Resume reading
});
28 What is a Buffer? When do you use it?
Intermediate
▼

A Buffer is a fixed-size chunk of memory allocated outside the V8 heap, used to store raw binary data.

When to use: Working with binary data — file I/O, network protocols, crypto, image processing.

JS// Create buffers
const buf1 = Buffer.from('Hello', 'utf-8');
const buf2 = Buffer.alloc(10); // 10 bytes, zero-filled
const buf3 = Buffer.allocUnsafe(10); // 10 bytes, NOT zero-filled (faster)

console.log(buf1.toString()); // 'Hello'
console.log(buf1.length);     // 5 bytes
console.log(buf1.toJSON());   // { type: 'Buffer', data: [72, 101, ...] }
⚠️ Security
Buffer.allocUnsafe() may contain old memory data. Use Buffer.alloc() for sensitive operations (Amex/BFSI context!).

30 How do you design retries/timeouts to avoid retry storms?
Advanced Amex
▼
  • Exponential backoff + jitter — Spread retries to avoid thundering herd
  • Max retries — Limit to 3-5 attempts
  • Circuit breaker — Stop calling failed dependency
  • Only retry idempotent operations — GET, PUT (NOT blind POST)
  • Strict timeouts — Fail fast, don't hold connections
JSasync function retryWithBackoff(fn, maxRetries = 3) {
  for (let i = 0; i <= maxRetries; i++) {
    try {
      return await fn();
    } catch (err) {
      if (i === maxRetries) throw err;
      const delay = Math.pow(2, i) * 1000 + Math.random() * 1000; // jitter!
      console.log(`Retry ${i + 1} in ${Math.round(delay)}ms`);
      await new Promise(r => setTimeout(r, delay));
    }
  }
}

Intermediate
▼

Node.js is built on the Reactor Pattern — a design pattern for non-blocking I/O:

  1. Application submits I/O request to the Event Demultiplexer
  2. Demultiplexer watches multiple I/O sources simultaneously
  3. When I/O completes, it pushes events to the Event Queue
  4. Event Loop processes the queue and invokes registered callbacks

This allows a single thread to handle thousands of concurrent connections efficiently.

34 How do you implement idempotency for payment APIs?
Must Know Amex
▼

Idempotency ensures repeated requests produce the same result — critical for payment systems.

JS// Idempotency middleware
async function idempotencyMiddleware(req, res, next) {
  const key = req.headers['idempotency-key'];
  if (!key) return res.status(400).json({ error: 'Idempotency-Key required' });

  // Check Redis for existing response
  const cached = await redis.get(`idempotency:${key}`);
  if (cached) {
    const response = JSON.parse(cached);
    return res.status(response.status).json(response.body);
  }

  // Capture the response
  const originalJson = res.json.bind(res);
  res.json = (body) => {
    // Cache for 24 hours
    redis.setex(`idempotency:${key}`, 86400, JSON.stringify({
      status: res.statusCode,
      body
    }));
    originalJson(body);
  };

  next();
}

// Usage
app.post('/api/payments', idempotencyMiddleware, paymentController.charge);
🏦 Why this matters at Amex
Network failures during payment processing can cause double charges. Idempotency keys prevent duplicate transactions on retries.

JSfunction myPromiseAll(promises) {
  return new Promise((resolve, reject) => {
    if (!Array.isArray(promises)) return reject(new TypeError('Not iterable'));
    if (promises.length === 0) return resolve([]);

    const results = [];
    let completed = 0;

    promises.forEach((p, index) => {
      Promise.resolve(p)
        .then(value => {
          results[index] = value; // Maintain order!
          completed++;
          if (completed === promises.length) resolve(results);
        })
        .catch(reject); // Fail-fast on first rejection
    });
  });
}

// Test

  Promise.resolve(1),
  new Promise(res => setTimeout(() => res(2), 100)),
  Promise.resolve(3)
]).then(console.log); // [1, 2, 3]

JSclass EventEmitter {
  constructor() {
    this.events = {};
  }

  on(event, listener) {
    if (!this.events[event]) this.events[event] = [];
    this.events[event].push(listener);
    return this;
  }

  emit(event, ...args) {
    if (!this.events[event]) return false;
    this.events[event].forEach(listener => listener(...args));
    return true;
  }

  off(event, listener) {
    if (!this.events[event]) return this;
    this.events[event] = this.events[event].filter(l => l !== listener);
    return this;
  }

  once(event, listener) {
    const wrapper = (...args) => {
      listener(...args);
      this.off(event, wrapper);
    };
    return this.on(event, wrapper);
  }
}

C3 Implement retry with exponential backoff + jitter

JSasync function retryWithBackoff(fn, options = {}) {
  const { maxRetries = 3, baseDelay = 1000, maxDelay = 30000 } = options;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (err) {
      if (attempt === maxRetries) throw err;

      const exponential = Math.pow(2, attempt) * baseDelay;
      const jitter = Math.random() * baseDelay;
      const delay = Math.min(exponential + jitter, maxDelay);

      console.log(`Attempt ${attempt + 1} failed. Retrying in ${Math.round(delay)}ms...`);
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }
}

// Usage
const data = await retryWithBackoff(() => fetch('https://api.amex.com/rates'), {
  maxRetries: 3,
  baseDelay: 1000
});
C4 Implement a Rate Limiter (sliding window)
Coding Amex

JSclass SlidingWindowRateLimiter {
  constructor(maxRequests, windowMs) {
    this.maxRequests = maxRequests;
    this.windowMs = windowMs;
    this.clients = new Map();
  }

  isAllowed(clientId) {
    const now = Date.now();
    const windowStart = now - this.windowMs;

    if (!this.clients.has(clientId)) {
      this.clients.set(clientId, []);
    }

    // Remove expired timestamps
    const timestamps = this.clients.get(clientId)
      .filter(ts => ts > windowStart);
    this.clients.set(clientId, timestamps);

    if (timestamps.length >= this.maxRequests) {
      return false; // Rate limited!
    }

    timestamps.push(now);
    return true;
  }
}

// Express middleware
const limiter = new SlidingWindowRateLimiter(100, 60 * 1000); // 100 req/min

app.use((req, res, next) => {
  const clientId = req.ip;
  if (!limiter.isAllowed(clientId)) {
    return res.status(429).json({ error: 'Too many requests' });
  }
  next();
});

JSasync function parallelLimit(tasks, concurrency) {
  const results = [];
  let index = 0;

  async function runNext() {
    while (index < tasks.length) {
      const currentIndex = index++;
      results[currentIndex] = await tasks[currentIndex]();
    }
  }

  // Create `concurrency` number of workers
  const workers = Array(Math.min(concurrency, tasks.length))
    .fill(null)
    .map(() => runNext());

  await Promise.all(workers);
  return results;
}

// Usage: Process 100 API calls, 5 at a time

const results = await parallelLimit(tasks, 5);

JSfunction debounce(fn, delay) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// Throttle (for comparison — also commonly asked)
function throttle(fn, limit) {
  let inThrottle = false;
  return function (...args) {
    if (!inThrottle) {
      fn.apply(this, args);
      inThrottle = true;
      setTimeout(() => (inThrottle = false), limit);
    }
  };
}

JSfunction promisify(fn) {
  return function (...args) {
    return new Promise((resolve, reject) => {
      fn(...args, (err, result) => {
        if (err) reject(err);
        else resolve(result);
      });
    });
  };
}

T1 What is the output? (Event Loop ordering)

1
6
4     ← nextTick (microtask, highest priority)
3     ← Promise (microtask)
2     ← setTimeout (timers phase)

Explanation: Synchronous code runs first (1, 6). Then microtasks: nextTick before Promises. Then macrotasks: setTimeout before setImmediate (outside I/O context, order can vary).

Quick Reference Cheat Sheet

Event Loop Priority

1. Sync code
2. process.nextTick()
3. Promise.then()
4. setTimeout(0)
5. setImmediate()

require vs import

require = CommonJS (sync, runtime)
import = ESM (async, static, tree-shakeable)

Buffer

Raw binary data, outside V8 heap.
alloc() = safe, allocUnsafe() = fast.
Use alloc() for security!

cluster vs worker_threads

cluster = multi-process (HTTP scale)
worker_threads = multi-thread (CPU work)

npm ci vs npm install

npm ci = clean install from lockfile (production, CI)
npm install = resolves from package.json

NODE_ENV=production

Enables Express view caching
Reduces verbose error output
Optimizes perf in many libs

Streams Types

Readable → Writable → Duplex → Transform
.pipe() handles backpressure automatically

Error Handling

Operational → Handle gracefully
Programmer → Crash & restart
Always: process.on('unhandledRejection')
